Project

Banking App

BANKING APP

Ruby lessons effectively took up only the first two weeks of the Coder Academy Fast-track bootcamp. After submitting a very successful first terminal application assignment with Matthew Puku, I decided I needed to consolidate and reflect on my own understanding on the programming principles covered. While I had no issues with following the lecture content, I struggled with the beast mode and morning challenges. So I decided to start a personal project into building a banking app on my own. Second to the ubiquitous to-do list app, a banking app is a typical application beginners would embark on when learning programming.


Functionalities

  • Make a deposit to an account
  • Make a withdrawal to an account
  • Switchboard menu interface accepting keyboard input
  • Basic error handling of user input (i.e. negative amounts)

With the guidance of Mr Matt Stubbs, teaching assistant at Coder Academy, I used the banking app project to gain a better understanding of the following

  • Method calls and passing arguments
  • Grouping functionalities by segmenting and planning functionalities around classes and objects
  • Looping arrays and hashes

Extensible Features

While this basic application is functional, it leaves a lot of room for future improvement and a litany of 'nice-to-have' features to be explored.

  • User input and menu interface should be encapsulated in a separate class of its own. Input sterilisation and user interaction can be kept separate away from the core calculations, data storage functionalities of the rest of the program for better extensibility and code maintenace. This is also in keeping with the SOLID and DRY principles of good coding practices.
  • I can experiment with inheritance by initialising an under-18, kids version of the account object, which can return higher interest rates on top of the balance.
  • It would be beneficial to build CRUD features to maintain a current history log of interactions with the application. This could include dated entries of the account balance, and changes to user authentication details. Ideally with each new user authenticated, a new directory could be instantiated which will contain .txt files as logs of balance changes.
  • Compound interest rates should be included in the account upkeep, along with account keeping fees. This can create more features that better align with everyday banking scenarios.
  • Functionalities and classes could be written into separate files for better readability and organisation.
  • I am not sure whether researching into gems would significantly boost the performance of an application as simple and basic as this. It may help to look into colorize or other display gems that could improve the user menu interface.

The codebase is freely viewable on my github repo. Please feel free to fork/clone or leave feedback as to how I can improve it.


The application was developed purely for educational purposes and does not use any copyrighted material
